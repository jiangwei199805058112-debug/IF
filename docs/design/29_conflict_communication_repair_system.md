# 沟通障碍、冲突升级与修复回应系统设计

本文档用于沉淀 IF 项目中的沟通障碍、冲突升级、积极倾听、精确表达、感受确认和修复回应机制。它不是普通沟通技巧笔记，而是用于支撑关系事件、NPC 反应、冲突分支、信任变化、旧伤记忆和关系报告的系统设计。

核心原则：

```text
亲密关系中的沟通问题不只是“有没有说”，而是：
是否准确表达、是否认真倾听、是否确认对方感受、是否把具体问题扩大成人格攻击，以及争执升温后能否暂停和修复。
```

本设计参考亲密关系中的沟通不良、抱怨方式、读心术、打断、批评、蔑视、防卫、石墙、积极倾听、知觉检验、第一人称陈述、XYZ 陈述、安慰表达和情绪确认等观点，并转译为 IF 的系统机制。本文档不收录原始截图或长篇原文，只保留可供项目使用的变量、事件模板、报告标签和落地边界。

---

## 1. 设计目的

此前文档已经沉淀了：

- `docs/design/25_attribution_memory_belief_system.md`：归因、记忆、关系信念、真实原因层和解释可信度；
- `docs/design/26_partner_perception_and_impression_system.md`：伴侣认知准确度、形象管理、危险知觉；
- `docs/design/27_communication_self_disclosure_system.md`：语言沟通、自我表露、秘密边界；
- `docs/design/28_questionnaire_communication_disclosure_module.md`：前期调查中的沟通表露模块。

本文档继续补足一个问题：

```text
当关系进入不满、抱怨、争吵、误解和情绪激活时，系统如何判断沟通是在修复关系，还是正在把问题推向升级？
```

IF 不应把争吵简单写成：

```text
吵架 = 扣分
不吵 = 加分
```

更合理的是：

```text
冲突结果 = 表达方式 + 倾听质量 + 感受确认 + 归因方式 + 防卫程度 + 暂停修复能力
```

有些争吵能暴露真实问题并推动修复；有些表面平静只是回避；有些沟通看似“说了很多”，其实一直在偏题、读心、反抱怨或人格攻击。

## 2. 系统一：沟通不良与冲突升级

### 2.1 设计原则

不幸福关系中的沟通不良往往不只是内容负面，而是沟通结构本身出了问题：

- 抱怨很多，但无法准确落到具体问题；
- 同时丢出太多问题，导致一个都解决不了；
- 读心式解释伴侣意图；
- 打断对方，转移话题；
- 用反向抱怨替代倾听；
- 批评人格，而不是描述具体行为；
- 蔑视、挖苦、羞辱；
- 防卫、找借口、反击；
- 拒绝回应或石墙；
- 最后进入敌对、冷冰冰、拒绝解决的状态。

### 2.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `communication_quality` | 当前沟通质量总评 |
| `off_beam_talk` | 偏离主题、从一个问题跳到另一个问题 |
| `kitchen_sinking` | 一次性翻出大量旧账，导致无法聚焦解决 |
| `mind_reading_tendency` | 读心术，擅自断定对方意图 |
| `negative_intent_attribution` | 把伴侣行为解释为恶意、故意或不在乎 |
| `interrupting_tendency` | 打断对方说话的倾向 |
| `cross_complaining` | 反向抱怨，用自己的抱怨回应对方抱怨 |
| `yes_butting_pattern` | 表面承认，随后用“但是”否定对方感受 |
| `defensive_response` | 防卫、找借口、拒绝承认影响 |
| `criticism_intensity` | 批评强度，尤其是人格和品德批评 |
| `contempt_signal` | 蔑视、讥讽、嘲笑、羞辱信号 |
| `stonewalling_level` | 石墙程度，拒绝回应或冷处理 |
| `belligerence_level` | 交战状态，挑衅式完全拒绝对方 |
| `conflict_escalation_risk` | 冲突升级风险 |

### 2.3 沟通不良事件模板

事件：玩家抱怨“你迟到让我很尴尬”。

NPC 可能回应：

| 回应 | 类型 | 效果 |
| --- | --- | --- |
| “你是说我迟到让你觉得不被重视，对吗？” | 复述/积极倾听 | 降低误解，给出修复机会 |
| “好吧，我以后注意。”但语气敷衍 | 表面赞同 | 短期降温，长期可能旧账累积 |
| “那你上次不也迟到？” | 反向抱怨 | 当前问题被转移，冲突升级 |
| “你就是喜欢小题大做。” | 人格批评 | 亲密和尊重下降 |
| “又来了，那你到底想怎样？” | 交战/挑衅 | 进入敌对循环 |
| 沉默、刷手机、不回应 | 石墙 | 对方更愤怒或更绝望 |

## 3. 系统二：精确表达与行为描述

### 3.1 设计原则

关系冲突中，表达越模糊、越人格化、越绝对化，越容易升级。表达越具体、越聚焦行为和影响，越容易修复。

错误方向：

```text
你总是这样。
你从来不听我说话。
你就是自私。
你根本不在乎我。
```

更可计算的表达应当包含：

```text
具体情境 + 具体行为 + 我的感受/影响 + 希望如何调整
```

### 3.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `behavior_description_accuracy` | 是否准确描述具体行为，而非泛化人格 |
| `global_blame_tendency` | 使用“总是、从来、根本”等绝对化指责 |
| `personality_attack_level` | 对人格、品德、性格的攻击程度 |
| `i_statement_skill` | 第一人称陈述能力 |
| `xyz_statement_skill` | XYZ 陈述能力 |
| `specific_request_clarity` | 是否提出具体可执行请求 |
| `emotion_ownership` | 是否承认“这是我的感受”，而非把情绪全责推给对方 |

### 3.3 第一人称陈述

第一人称陈述的作用是让表达从指责转为自我感受说明。

结构：

```text
我感到……
因为……
我希望……
```

示例：

```text
我刚刚被打断时感到很生气，
因为我觉得自己没有被认真听完，
我希望你先让我说完，然后我们再讨论你的想法。
```

### 3.4 XYZ 陈述

XYZ 陈述用于更精确地描述冲突。

结构：

```text
当你在 Y 情境下做 X 的时候，我感到 Z。
```

示例：

```text
当我们在讨论约会安排时，你突然开始看手机，我感到自己被忽视。
```

### 3.5 游戏事件模板

事件：对方打断你说话。

玩家可选表达：

| 选项 | 类型 | 效果 |
| --- | --- | --- |
| “你怎么这么不为我着想！” | 泛化攻击 | 冲突升级 |
| “你刚才打断我，我很生气。” | 第一人称陈述 | 对方更容易回应 |
| “当我说到这件事时你打断我，我觉得自己没有被听见。” | XYZ 陈述 | 最有利于修复 |
| “算了，没事。” | 压抑回避 | 短期降温，长期旧账上升 |
| “你也别说了。” | 反击 | 防卫和敌意上升 |

## 4. 系统三：安慰、同情与关心表达

### 4.1 设计原则

安慰不是提供漂亮话，也不是强行乐观。错误安慰可能伤害对方，因为它会否定对方正在经历的痛苦。

危险安慰方向：

- 过早要求对方忘记；
- 用乐观话术压过痛苦；
- 对痛苦进行比较或贬低；
- 给出空洞建议；
- 暗示对方不应该这么难过。

有效安慰更接近：

```text
承认事实 + 承认痛苦 + 表达关心 + 不急着评价 + 不抢走对方感受
```

### 4.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `comfort_skill` | 安慰能力 |
| `pain_acknowledgment` | 是否承认对方痛苦 |
| `premature_reassurance_risk` | 过早安慰、强行乐观的风险 |
| `invalidating_comfort` | 否定式安慰 |
| `advice_overload` | 对方需要被理解时却给太多建议 |
| `grief_support_quality` | 面对丧失、失败、严重压力时的支持质量 |
| `supportive_silence` | 能否以陪伴式沉默支持对方 |

### 4.3 安慰事件模板

事件：NPC 失去亲人或经历重大失败。

玩家回应：

| 回应 | 类型 | 效果 |
| --- | --- | --- |
| “我很抱歉。你现在一定很痛。” | 承认痛苦 | 支持质量上升 |
| “你愿意说的话，我在这里听。” | 陪伴式支持 | 安全感上升 |
| “早点忘了吧。” | 过早要求恢复 | 对方感到被否定 |
| “你还有很多好日子。” | 强行乐观 | 可能适得其反 |
| “至少情况没那么糟。” | 贬低痛苦 | 受伤和距离上升 |
| “你应该这样做……” | 建议过载 | 对方可能觉得不被理解 |

## 5. 系统四：积极倾听与复述

### 5.1 设计原则

积极倾听有两个任务：

1. 准确理解对方话里的意思；
2. 让对方感到自己被认真听见。

复述不是机械重复，而是用自己的话确认理解。

### 5.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `active_listening_skill` | 积极倾听能力 |
| `paraphrasing_skill` | 复述能力 |
| `perception_checking_skill` | 知觉检验能力 |
| `listening_patience` | 听完对方说话的耐心 |
| `premature_conclusion_risk` | 过早下结论风险 |
| `response_latency_tolerance` | 面对情绪信息时是否能停顿、确认再回应 |

### 5.3 复述模板

结构：

```text
你是说……，对吗？
我理解的是……，有没有理解错？
你刚刚最在意的是……，是这样吗？
```

例子：

```text
你是说你不是不欢迎我妈，而是最近课程压力很大，希望下周家里不要再安排太多事，对吗？
```

### 5.4 知觉检验模板

知觉检验用于检查自己的推断是否准确，而不是直接读心。

错误方向：

```text
你就是不想见我家人。
```

知觉检验：

```text
你刚才听到我妈要来时好像有点不安，是因为最近太忙，还是因为你对她来家里本身有压力？
```

## 6. 系统五：守礼而坚定与暂停机制

### 6.1 设计原则

在争吵升级时，最难做到的是继续清晰表达、认真倾听、保持礼貌和镇定。系统不应要求玩家永远冷静，但应允许玩家通过暂停机制阻断恶性循环。

### 6.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `civil_assertiveness` | 礼貌而坚定的表达能力 |
| `conflict_timeout_skill` | 暂停争执的能力 |
| `physiological_flooding` | 情绪或生理激活导致无法思考 |
| `cooldown_repair_window` | 冷静后回到问题的修复窗口 |
| `revenge_reply_impulse` | 被激怒后反击冲动 |
| `repair_after_timeout` | 暂停后是否回来继续解决问题 |

### 6.3 暂停模板

可用表达：

```text
我现在太生气了，怕继续说会伤人。给我十分钟冷静一下，我们等会儿继续谈。
```

暂停不是冷处理。有效暂停必须满足：

1. 说明自己需要冷静；
2. 说明不是逃避问题；
3. 给出明确回来继续谈的时间或条件；
4. 之后真的回来处理问题。

## 7. 系统六：尊重与确认

### 7.1 设计原则

确认不等于赞同。一个人可以不认同伴侣的观点，但仍然承认对方感受的合理性。

错误方向：

```text
你说得对，我以后都听你的。
```

这可能是敷衍、讨好或压抑。

更好的确认：

```text
我明白你为什么会难受。虽然我对这件事的看法不完全一样，但我愿意认真听你说。
```

### 7.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `validation_skill` | 感受确认能力 |
| `respectful_disagreement` | 尊重地不同意 |
| `partner_view_acknowledgment` | 是否承认对方观点有其理由 |
| `emotional_legitimacy_grant` | 是否承认对方情绪有其合理性 |
| `dismissive_disagreement` | 轻视式不同意 |
| `appeasement_without_understanding` | 表面赞同但没有真正理解 |

### 7.3 事件模板

事件：对方说“我讨厌你那样做”。

玩家回应：

| 回应 | 类型 | 效果 |
| --- | --- | --- |
| “好吧，你说得对，我不做了。” | 表面赞同 | 短期降温，但未必修复 |
| “你又来了，真麻烦。” | 蔑视 | 冲突升级 |
| “我明白你不舒服。你说的有道理，但我也想说明我的感受。” | 确认 + 自我表达 | 有利于修复 |
| “你凭什么要求我？” | 防卫/反击 | 敌意上升 |

## 10. 报告标签设计

| 标签 ID | 含义 | 报告表达方向 |
| --- | --- | --- |
| `precise_expressor` | 精确表达者 | 更能描述具体行为和感受，减少人格攻击 |
| `mind_reading_pattern` | 读心倾向 | 容易擅自判断对方意图，需要知觉检验 |
| `kitchen_sinking_pattern` | 旧账堆叠 | 容易一次性翻出太多问题，导致无法解决当前问题 |
| `interrupting_pattern` | 打断倾向 | 情绪上来时容易打断对方，影响被听见感 |
| `defensive_responder` | 防卫回应者 | 面对抱怨时容易先解释、反击或找借口 |
| `stonewalling_pattern` | 石墙模式 | 压力过高时倾向沉默、撤退或拒绝回应 |
| `validation_capable` | 能确认感受 | 即使不同意，也能承认对方感受有其理由 |
| `timeout_repairer` | 暂停修复者 | 能在情绪过高时暂停，并回来继续谈 |
| `contempt_risk` | 蔑视风险 | 可能用讥讽、嘲笑或人格贬低破坏尊重感 |
| `active_listener` | 积极倾听者 | 能复述、澄清和检查理解，减少误解 |

## 11. 事件库方向

后续可以设计以下事件：

1. 对方抱怨你迟到；
2. 玩家抱怨对方总是玩手机；
3. 一方说起家庭压力，另一方急着给建议；
4. 一方说“你从来不听我说话”；
5. 对方沉默不回应；
6. 玩家忍不住翻旧账；
7. NPC 用反向抱怨回应玩家；
8. 玩家尝试用 XYZ 陈述表达不满；
9. NPC 在争吵中请求暂停；
10. 一方用嘲讽语气说“那又怎样”。

每个事件都应区分：

```text
真实问题是什么？
表达是否准确？
对方是否听懂？
是否存在人格攻击？
是否有感受确认？
是否进入防卫或石墙？
是否留下修复窗口？
```

## 12. 不做什么

本系统暂不做：

1. 不要求玩家永远理性；
2. 不把争吵直接等同于关系失败；
3. 不把沉默一律视为冷暴力，需要结合情境判断；
4. 不把所有抱怨都视为攻击；
5. 不把道歉或赞同一律视为修复；
6. 不把技巧写成固定加分按钮；
7. 不做心理诊断；
8. 不上传原始截图或大段原文；
9. 不接 AI API。

## 13. 一句话总结

```text
IF 的冲突沟通系统要模拟的不是“谁对谁错”，而是：
伴侣如何在不满中表达、倾听、确认、暂停和修复；以及他们如何因为读心、批评、蔑视、防卫和石墙而把可解决的问题推向关系伤害。
```
