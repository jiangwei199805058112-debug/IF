# v0.1 MVP 实施计划

本文件说明什么时候才进入代码阶段，以及第一版代码只实现什么。当前提交仍然不写代码。

## 1. 进入代码阶段的条件

只有满足以下条件，才开始写代码：

- 核心规则母版完成。
- 关系引擎规则完成。
- 14天流程完成。
- 3个种子事件分支表完成。
- 数据结构蓝图完成。
- 样例角色完成。
- 纸面测试用例完成。

进入代码阶段前，还应确认：

- v0.1 仍只做 14 天测试版。
- 不新增几十个事件。
- 不做 UI、数据库和 AI 生成。
- 所有亲密相关机制只保留成年人系统规则，不写露骨内容。

## 2. 第一版代码目标

第一版代码只做：

- 控制台文字原型。
- 选择主角。
- 选择开局模式。
- 选择一个样例角色组合。
- 跑 14 天。
- 触发 3 个种子事件。
- 输出模糊感知反馈。
- 记录记忆账本。
- 第14天输出阶段结算。

第一版目标不是做完整游戏，而是验证规则闭环能否被最小实现跑通。

## 3. 第一版不做内容

第一版不做：

- UI。
- 图片。
- 存档。
- AI生成。
- 完整问卷。
- 完整经济系统。
- 完整亲密系统。
- 完整第三者系统。
- 完整一年模拟。
- 数据库。
- 网络服务。
- 多结局演出。

## 4. 推荐项目结构

这里只写建议，不创建代码文件：

```text
if_game/
  main.py
  models.py
  engine.py
  event_loader.py
  data/
    sample_characters.json
    seed_events.json
    day_flow.json
tests/
  smoke_test.py
```

建议职责：

- `main.py`：控制台入口，处理选择和 14 天流程。
- `models.py`：定义数据对象。
- `engine.py`：执行关系结算、感知反馈和阶段判定。
- `event_loader.py`：加载事件和样例角色配置。
- `sample_characters.json`：样例角色组合。
- `seed_events.json`：3 个种子事件和分支。
- `day_flow.json`：14 天节奏表。
- `smoke_test.py`：最小启动和流程测试。

## 5. 配置文件优先原则

第一版应优先使用配置文件承载内容：

- 角色样例放配置。
- 事件分支放配置。
- 14 天节奏放配置。
- 行为标签和后果方向放配置或常量。

原因：

- 文档仍在快速调整，配置比硬编码更容易维护。
- 事件扩展时不需要频繁改引擎。
- 纸面测试用例可以和配置字段对照。

禁止把分支结果写成固定“选项 A 好感 +5”。即使未来映射成数值，也应先经过行为标签、上下文和角色维度。

## 6. 最小控制台原型流程

建议流程：

1. 启动控制台原型。
2. 选择玩家角色或使用默认玩家。
3. 选择开局模式：正在聊天、暧昧中、刚恋爱。
4. 选择样例角色组合 A-E。
5. 初始化 `RelationshipState`、`PerceivedRelationshipState`、`BoundaryAgreement` 和 `MemoryEntry` 空列表。
6. 从第 1 天跑到第 14 天。
7. 每天按早上、白天、晚上、深夜展示关键文本。
8. 在指定天数触发 `MSG_001`、`SOC_001`、`CONFLICT_001`。
9. 玩家选择一个选项。
10. 系统根据分支、行为标签、性格、压力和记忆结算方向性后果。
11. 输出模糊感知反馈，不展示真实值。
12. 写入必要记忆。
13. 第 7 天输出中期反馈。
14. 第 14 天输出阶段结算。

## 7. 最小数据文件

第一版最小数据文件建议：

| 文件 | 内容 | 来源文档 |
| --- | --- | --- |
| `sample_characters.json` | 5 组样例角色和性格维度 | `10_sample_character_profiles.md` |
| `seed_events.json` | 3 个种子事件、分支和选项 | `05_seed_events.md`、`08_seed_event_branch_tables.md` |
| `day_flow.json` | 14 天默认节奏、必触发事件、时间段 | `07_v0_1_14_day_flow.md` |
| `behavior_tags.json` | 行为标签定义 | `06_relationship_engine_rules.md` |
| `schema_notes.json` | 对象字段最小版本 | `09_data_schema_blueprint.md` |

第一版可以先只创建前三个数据文件；后两个可作为后续整理。

## 8. 最小 smoke test

未来 smoke test 应该检查：

- 能启动。
- 能加载样例角色。
- 能跑 14 天。
- 能触发 `MSG_001`、`SOC_001`、`CONFLICT_001`。
- 能生成感知反馈。
- 能写入记忆。
- 能输出第14天结算。

建议 smoke test 覆盖：

- 组合A 跑出轻微不安或冷淡。
- 组合E 跑出有效修复或升温。
- 至少一次 `MemoryEntry` 被写入。
- 玩家输出中不出现真实内部数值。

## 9. 提交与验证规则

进入代码阶段后的提交规则：

- 不使用 `git add .` 盲目提交。
- 小步提交，每次只提交明确范围。
- 提交前运行 `git status --short`。
- 提交前运行 `git diff --check`。
- 代码提交必须运行最小 smoke test。
- 文档变更可以不跑代码测试，但要说明未运行原因。
- 禁止 force push。
- 禁止 `git reset --hard`，除非用户明确要求。

第一版代码提交应优先包含：

- 数据模型。
- 配置加载。
- 14 天流程。
- 3 个事件触发。
- smoke test。

## 10. 后续扩展顺序

建议扩展顺序：

1. 把 v0.1.2 文档字段整理成最小 JSON 配置。
2. 实现控制台 14 天流程。
3. 实现 3 个种子事件分支选择。
4. 实现感知反馈和记忆写入。
5. 实现第 14 天阶段结算。
6. 跑纸面测试用例对照 smoke test。
7. 扩展到 20 个事件。
8. 增加完整问卷输入。
9. 增加更细的经济、第三者和亲密规则。
10. 再评估是否需要 UI、存档、AI 解析或数据库。

任何扩展都应先验证当前 14 天闭环没有破坏，再增加复杂度。
