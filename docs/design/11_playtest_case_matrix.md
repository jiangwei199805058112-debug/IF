# 纸面测试用例矩阵 v0.1

本文件用于在写代码前检查规则是否闭环。测试用例只验证纸面规则，不运行程序。

## 1. 测试目标

- 验证 3 个种子事件是否能从真实分支推导到感知反馈、记忆和后续钩子。
- 验证 14 天流程是否能输出不同阶段结算。
- 验证真实值与感知值偏差是否可理解。
- 验证撒谎、隐瞒、破绽和暴露是否形成闭环。
- 发现规则缺口后，先修正文档，再进入代码阶段。

## 2. 测试字段说明

每个测试用例至少包含：

| 字段 | 说明 |
| --- | --- |
| 测试ID | 用例唯一编号 |
| 测试目标 | 本用例要验证什么 |
| 开局模式 | 正在聊天 / 暧昧中 / 刚恋爱 |
| 角色组合 | 使用 `10_sample_character_profiles.md` 的组合 |
| 前置状态 | 压力、记忆、边界、阶段等 |
| 触发事件 | 事件 ID |
| 真实分支 | 分支 ID 或真实情况 |
| 玩家选择 | 玩家行为 |
| NPC解释方式 | NPC 的解释或反应 |
| 预期真实后果 | 内部方向性结果 |
| 预期感知反馈 | 玩家看到的模糊反馈 |
| 预期记忆写入 | 是否写入记忆账本 |
| 预期后续钩子 | 后续可能触发内容 |
| 是否通过 | 纸面测试时填写 |
| 备注 | 规则缺口或修正建议 |

## 10. 通过标准

一个纸面测试用例通过，需要满足：

- 能明确找到触发事件和真实分支。
- 玩家选择能映射到行为标签。
- NPC 解释方式符合性格、压力和前置状态。
- 预期真实后果使用方向描述，而不是固定数值。
- 感知反馈不暴露真实值。
- 记忆写入规则明确。
- 后续钩子能接到 14 天流程或阶段结算。
- 高风险行为有合理后果。

## 11. 失败后如何修正规则

若测试失败，按以下顺序修正：

1. 如果字段缺失，先更新 `09_data_schema_blueprint.md`。
2. 如果角色无法触发差异，更新 `10_sample_character_profiles.md`。
3. 如果事件分支不清，更新 `08_seed_event_branch_tables.md`。
4. 如果后果无法结算，更新 `06_relationship_engine_rules.md`。
5. 如果 14 天节奏不顺，更新 `07_v0_1_14_day_flow.md`。
6. 修正后重新跑同一纸面用例，不直接跳到写代码。
