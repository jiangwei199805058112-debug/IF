# 伴侣认知准确度、形象管理与危险知觉系统设计

本文档用于补充 IF 项目中的亲密关系认知机制。它不是普通恋爱心理学资料整理，而是用于支撑后续关系解释、问卷报告、NPC 行为理解、信任校准和长期关系变化的系统设计。

核心原则：

```text
亲密关系中的“我了解你”，不等于真实了解。
伴侣认知既可能因为长期相处而更准确，也可能因为投射、旧印象、理想化、危险扫描、选择性忽视和自我实现预言而更加失真。
```

本设计参考亲密关系中伴侣认知、形象管理、知觉准确度、危险知觉和知觉者影响等社会心理学观点，并转译为 IF 的系统机制。本文档不收录原始截图或长篇原文，只沉淀可供项目使用的机制、变量、事件模板和报告边界。

---

## 1. 设计目的

IF 当前已经具备：

- 问卷建档；
- 关系倾向判断；
- 真实原因层；
- 解释可信度；
- 可见线索；
- 信任校准；
- 关系事件解释原型。

本系统要继续补足一个问题：

```text
当玩家或 NPC 说“我了解你”“我知道你在想什么”“你就是这样的人”时，
系统应该如何判断这种认知是否准确？
```

现实亲密关系中，人对伴侣的判断并不总是稳定准确。关系越亲密，信息量通常越多，但偏差也可能越强。IF 不应简单认为：

```text
相处越久 → 越了解对方 → 解释越准确
```

更合理的是：

```text
认知准确度 = 相处信息量 + 观察动机 + 特质可见度 + 观察者能力
             - 投射偏差 - 旧印象固化 - 理想化 - 危险扫描 - 选择性忽视
```

## 2. 系统一：伴侣认知准确度

### 2.1 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `partner_model_accuracy` | 实际理解伴侣的准确度 |
| `partner_model_confidence` | 自认为了解伴侣的信心 |
| `partner_knowledge_gap` | 自信程度与真实准确度之间的落差 |
| `projection_bias` | 把自己的想法、需求、恐惧投射到伴侣身上的程度 |
| `outdated_partner_model` | 用旧印象理解现在的伴侣 |
| `idealized_partner_model` | 把伴侣理解成自己希望的样子 |
| `negative_partner_model` | 把伴侣理解成更糟、更自私或更危险的人 |
| `trait_observability` | 某种特质是否容易通过日常行为被观察到 |
| `perception_motivation` | 是否有动力真正理解对方 |
| `perception_skill` | 观察、共情、整合线索的能力 |

### 2.2 关键判断

玩家或 NPC 对伴侣的认知可以分为四种状态：

| 自信程度 | 实际准确度 | 状态 |
| --- | --- | --- |
| 高 | 高 | 稳定理解 |
| 高 | 低 | 自信误读 |
| 低 | 高 | 敏感但不确定 |
| 低 | 低 | 混乱猜测 |

系统不应只看“我很确定”，而要同时看：

```text
他说得有多确定？
他说得有多准？
这种判断来自证据、经验、投射、旧账，还是恐惧？
```

### 2.3 游戏表现

同一事件中，不同玩家会产生不同理解。

例子：对方今天语气冷淡。

| 玩家状态 | 可能解释 |
| --- | --- |
| 高准确 + 稳定信任 | “可能是累了，但这次确实有点异常。” |
| 高自信 + 高投射 | “他肯定是不爱我了。” |
| 低自信 + 高危险扫描 | “我不知道是不是我想多了，但我很不安。” |
| 高旧印象固化 | “他以前一冷淡就是逃避，所以这次肯定也是。” |
| 高选择性忽视 | “没事，他只是忙。” |

## 3. 系统二：形象管理变化

### 3.1 设计原则

亲密关系中，人不会永远维持暧昧期或恋爱初期的形象管理。随着关系推进，伴侣可能更放松，也可能减少投入。

因此，形象变化不能简单等同于：

```text
变随便 = 不爱了
```

也不能简单等同于：

```text
变随便 = 更信任了
```

它需要结合真实原因层、关系阶段、投入变化、压力状态和历史表现判断。

### 3.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `self_presentation_effort` | 形象维持努力 |
| `romantic_effort_level` | 恋爱经营努力 |
| `romantic_effort_decay` | 恋爱经营衰减 |
| `domestic_comfort_level` | 亲密后的松弛程度 |
| `domestic_image_gap` | 亲密前后形象落差 |
| `effort_expectation_gap` | 一方期待的经营程度与另一方实际付出的落差 |
| `taken_for_granted_signal` | “反正你不会离开”的信号 |
| `stress_based_effort_drop` | 因压力、疲惫、现实负担导致的经营下降 |
| `commitment_based_relaxation` | 因关系稳定而自然放松 |

### 3.3 关系阶段变化

| 阶段 | 常见表现 |
| --- | --- |
| 刚认识 | 高形象管理，高自我包装 |
| 暧昧期 | 高回应、高表现、高吸引力维护 |
| 刚恋爱 | 仍努力经营，但开始暴露真实习惯 |
| 稳定关系 | 松弛上升，形象管理下降 |
| 关系疲惫期 | 浪漫经营下降，敷衍感增强 |
| 冲突期 | 形象管理可能转为防御、冷淡或表演性补偿 |

### 3.4 事件模板

事件：对方现在见你越来越随便。

真实原因可能是：

1. 关系稳定后更放松；
2. 最近压力大，没有精力维持状态；
3. 对关系投入下降；
4. 觉得你不会离开，所以减少经营；
5. 对方本来就是低形象管理人格；
6. 对方在别处投入更多形象管理。

玩家解释可能是：

1. “他/她信任我了。”
2. “他/她不重视我了。”
3. “是不是不爱了？”
4. “我是不是要求太多？”
5. “是不是把精力给别人了？”
6. “这才是真实的他/她。”

系统要判断：

```text
解释是否有证据支持？
是否存在前后对比？
是否存在选择性投入？
是否存在第三方或社交场合中的形象差异？
```

## 4. 系统三：危险知觉

### 4.1 设计原则

亲密关系中的危险知觉有双重作用：

```text
它可以保护人，让人更早发现真实风险；
也可以消耗人，让人对模糊线索过度解释。
```

IF 不应把敏感者都写成“疑心病”，也不应把信任者都写成“成熟”。敏感可能是准确警觉，信任也可能是选择性失明。

### 4.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `threat_attention_bias` | 对潜在危险信号的注意偏向 |
| `relationship_threat_sensitivity` | 对关系威胁的敏感度 |
| `danger_perception_accuracy` | 危险判断准确度 |
| `over_suspicion_tendency` | 过度怀疑倾向 |
| `selective_blindness` | 对不想看到的问题选择性忽视 |
| `protective_avoidance` | 为保护关系或情绪而回避真相 |
| `evidence_tolerance_threshold` | 需要多少证据才愿意相信问题存在 |
| `ambiguity_distress` | 面对暧昧线索时的痛苦程度 |
| `threat_confirmation_seeking` | 主动寻找危险证据的倾向 |
| `reassurance_dependency` | 依赖对方安抚来降低不安的程度 |

### 4.3 四象限校准

| 判断 | 事实 | 类型 |
| --- | --- | --- |
| 怀疑有问题 | 真的有问题 | 准确警觉 |
| 怀疑有问题 | 实际没问题 | 误会/焦虑 |
| 相信没问题 | 实际没问题 | 稳定信任 |
| 相信没问题 | 真的有问题 | 被欺骗/低估风险 |

系统必须保留这四种可能，不能把所有怀疑归为焦虑，也不能把所有信任归为成熟。

### 4.4 事件模板

事件：对方说没看手机，但期间更新了社交媒体。

真实原因可能是：

1. 真的没注意到你的消息，但刷了社交媒体；
2. 看到了，但觉得不急；
3. 不想面对你的问题；
4. 对你优先级下降；
5. 正在和别人互动；
6. 故意冷处理；
7. 撒谎。

玩家解释可能是：

1. “他/她只是没注意。”
2. “他/她对别人有空，对我没空。”
3. “他/她不想理我。”
4. “他/她在骗我。”
5. “我是不是太敏感了？”

证据层应检查：

- 是否已读；
- 是否多次发生；
- 是否只对玩家不可用；
- 是否存在解释变化；
- 是否删除痕迹；
- 是否有第三方线索；
- 是否与此前承诺冲突。

## 5. 系统四：知觉者对伴侣的反向影响

### 5.1 设计原则

玩家或 NPC 不是被动观察者。一个人如何理解伴侣，会影响自己的行为，而自己的行为又会塑造伴侣的后续反应。

核心链路：

```text
我如何看你 → 我如何对你 → 你如何被影响 → 你是否真的开始变化
```

### 5.2 核心变量

| 变量 ID | 含义 |
| --- | --- |
| `perceiver_influence` | 观察者对伴侣的反向塑造影响 |
| `positive_partner_shaping` | 正向期待带来的鼓励和支持 |
| `negative_partner_shaping` | 负面期待带来的压迫和防御 |
| `self_fulfilling_relationship_loop` | 自我实现预言循环 |
| `trust_based_growth` | 被信任后更愿意负责 |
| `suspicion_based_withdrawal` | 被怀疑后更想逃避或隐瞒 |
| `control_reactance` | 被控制后产生反抗 |
| `ideal_pressure` | 被理想化后产生压力 |
| `disappointment_loop` | 期待落空后的反复失望循环 |

### 5.3 游戏表现

| 玩家认知 | 玩家行为 | 伴侣可能变化 |
| --- | --- | --- |
| 认为对方可靠 | 给机会、鼓励、信任 | 对方更愿意负责 |
| 认为对方会背叛 | 监控、质问、试探 | 对方更想隐瞒或逃避 |
| 认为对方不在乎 | 冷处理、撤退 | 对方也减少投入 |
| 认为对方有潜力 | 支持、反馈、共建 | 对方更接近理想形象 |
| 认为对方没救 | 贬低、控制、放弃 | 关系进入恶性循环 |
| 过度理想化对方 | 不允许对方暴露缺点 | 对方压力上升或开始表演 |

## 6. 报告标签设计

### 6.1 标签边界原则

报告标签不能道德审判玩家或 NPC。标签应描述倾向、风险和适配方式，而不是给人下定性结论。

错误写法：

```text
你疑心病很重。
你恋爱脑。
你很容易被骗。
你控制欲太强。
```

推荐写法：

```text
你对关系中的异常信号较敏锐，这能帮助你发现真实风险，但也可能让你在证据不足时承受较高压力。
```

### 6.2 可用标签

| 标签 ID | 含义 | 报告表达方向 |
| --- | --- | --- |
| `confident_misreader` | 自信误读者 | 对伴侣判断很有信心，但部分判断可能来自旧印象或投射 |
| `accurate_alertness` | 准确警觉 | 对异常信号敏锐，且部分判断可能接近事实 |
| `over_suspicion_pattern` | 过度怀疑模式 | 容易在证据不足时提前进入威胁解释 |
| `selective_blindness_pattern` | 选择性忽视模式 | 为维持关系稳定，可能低估不一致信号 |
| `romantic_idealizer` | 浪漫理想化 | 更容易用理想化方式解释伴侣行为 |
| `outdated_partner_model` | 旧印象固化 | 容易用过去的伴侣形象理解现在的伴侣 |
| `threat_scanner` | 危险扫描者 | 会优先扫描潜在风险，保护性强但消耗较高 |
| `reassurance_seeker` | 安抚需求者 | 需要较明确的回应和确认来稳定关系安全感 |
| `low_effort_sensitive` | 低投入敏感 | 对伴侣经营下降、回应变少、态度变随便较敏感 |
| `stable_interpreter` | 稳定解释者 | 能结合证据、关系历史和当下情境综合判断 |
| `comfort_as_trust_reader` | 将松弛解读为信任 | 倾向于把伴侣放松、随便理解为关系稳定 |
| `effort_decay_detector` | 投入衰减识别者 | 对关系经营下降有较高识别能力 |

## 7. 与真实原因层的关系

本系统不替代真实原因层，而是补充“认知准确度”。

真实原因层回答：

```text
事情真实是什么？
对方真实意图是什么？
伤害程度如何？
```

解释可信度系统回答：

```text
对方给出的解释可信吗？
可见线索是否支持该解释？
```

伴侣认知系统回答：

```text
玩家或 NPC 为什么会这样理解？
这种理解是准确、误会、投射、旧印象、危险扫描，还是选择性忽视？
```

三者合并后，才能形成完整关系解释：

```text
真实发生了什么
→ 对方如何解释
→ 玩家看到了什么
→ 玩家如何理解
→ 玩家理解是否准确
→ 玩家反应如何影响后续关系
```

## 9. 不做什么

本系统暂不做：

1. 不做心理诊断；
2. 不给玩家或 NPC 贴病理标签；
3. 不把敏感等同于错误；
4. 不把信任等同于成熟；
5. 不把形象管理下降直接等同于不爱；
6. 不把长期相处直接等同于准确了解；
7. 不让单一事件决定人格或关系结局；
8. 不写成固定扣分系统；
9. 不使用 AI API；
10. 不要求 UI 复杂化。

## 11. 一句话总结

```text
IF 中的伴侣认知系统要模拟的不是“你是否了解对方”，而是：
你有多自信地理解对方、这种理解有多准确、它来自证据还是投射，以及你的理解会怎样反过来塑造这段关系。
```
