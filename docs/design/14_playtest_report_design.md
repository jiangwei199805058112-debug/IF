# 试玩报告与结局解释设计 v0.1

## 1. 设计目标

v0.1.7 的试玩报告要从开发调试日志转成玩家能读懂的关系复盘。报告仍然基于 14 天控制台原型，不新增事件库、不接 AI、不展示真实内部数值。

## 2. 当前问题

v0.1.6 的报告直接展示 `scenario_id`、`feedback_level`、`active_hooks` 等字段。这些字段适合开发排查，但玩家很难理解它们和关系走向的关系，也容易把测试场景误读成固定剧情。

## 3. 玩家可读报告结构

报告结构：

```text
# IF 关系复盘报告

场景：有效修复路径
结局：确认关系
副标签：有效修复

关系概述：
这 14 天里，你们遇到过消息延迟和边界确认，但多数问题都被及时解释或修复。

关键转折点：
1. 第 3 天，对方消息延迟后主动解释。
2. 第 8 天，异性饭局提前说明并说清边界。
3. 第 12 天，冲突后双方愿意把问题摆到台面上。

主要原因：
- 信息缺口被及时补上。
- 边界没有靠猜，而是被明确表达。
- 冲突没有拖成冷战。

后续机会：
- 可以继续建立稳定联系节奏。
- 可以进一步讨论异性社交边界。
```

## 4. 主结局与副标签

主结局表示当前阶段，v0.1.7 支持：

- 确认关系
- 升温
- 继续暧昧
- 暧昧降温
- 冷淡
- 分手危机
- 分手

副标签表示关系模式，v0.1.7 支持：

- 有效修复
- 冷处理循环
- 信任受损
- 边界未清
- 重大谎言
- 分分合合倾向
- 现实压力
- 关系仍有吸引

“分分合合倾向”更适合作为副标签，不总是主结局。出现重大谎言、前任隐瞒或情绪性拉黑时，主结局优先显示“分手危机”。

## 5. 关键转折点

关键转折点解释关系在哪些节点改变方向。它不复述全部 transcript，而是提取影响关系走向的事件，例如消息解释是否补足、异性边界是否说清、冲突后是否冷战或修复。

## 6. 主要原因

主要原因优先来自记忆账本，因为记忆账本记录的是 14 天内真正留下痕迹的互动。原因只使用玩家可见的文字，不输出信任、安全感、冲突等真实数值。

## 7. 后续隐患

后续隐患把开发钩子翻译成关系风险，例如回复模式、边界谈判、冷处理、复联沟通和分手谈话。隐患用于复盘，不代表后续必然发生。

## 8. 修复机会

修复机会说明下一轮测试可以尝试的沟通方向，例如建立忙碌时补充说明的习惯、把异性社交边界具体说清、或在复联时谈清楚是否还会切断联系。

## 9. 开发字段与玩家文本映射

| 开发字段/钩子 | 玩家文本 |
|---|---|
| `stable` | 关系整体稳定 |
| `slight_unease` | 有轻微不安 |
| `obvious_abnormal` | 关系里出现明显不对劲 |
| `relationship_crisis` | 已接近关系危机 |
| `reply_pattern_watch` | 回复模式可能继续引发不安 |
| `boundary_talk` | 异性边界还需要谈清 |
| `cold_war` | 冷处理模式已经出现 |
| `reply_slowdown` | 回复变慢可能成为隐患 |
| `crisis` | 关系进入危机线 |
| `breakup_talk` | 后续可能进入分手谈话 |
| `repair_success` | 这次冲突完成了有效修复 |
| `reconnect_talk` | 复联沟通仍有机会 |

## 10. v0.1.7 启用范围

v0.1.7 只用于现有 14 天控制台原型和 5 条内置测试场景。它不新增大量剧情、不扩大事件库、不做 UI、不接 AI，也不把测试场景写成玩家固定剧情路线。
